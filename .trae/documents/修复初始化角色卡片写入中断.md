## 现状链路梳理
- 入口是管理界面的“初始化”按钮：[chroma_admin.py:_on_init_db/_init_db_stepwise](file:///d:/SuperAntAgent/ui/chroma_admin.py#L514-L574)
- 每个角色依次执行：
  - 1) ensure_ready：`/call method=ensure_ready` → [server.py:/call](file:///d:/SuperAntAgent/chromaserver/server.py#L204-L231) → [service.py:VectorStoreWorker.call](file:///d:/SuperAntAgent/chromaserver/service.py#L137-L208)
  - 2) preload_system_prompts 勾选时：把 system_prompts.yaml 分块后，调用 `record_to_memory(thinking=system_prompts_init, content=blocks, memory_type=personal)`：[chroma_admin.py](file:///d:/SuperAntAgent/ui/chroma_admin.py#L568-L572)
- 服务端 `record_to_memory` 会走 ReMe 的“个人记忆总结流”：[reme_config.yaml:summary_personal_memory](file:///d:/SuperAntAgent/configs/reme_config.yaml#L55-L71)
  - 该流先 InfoFilter，把“用户信息”打分后过滤；系统提示词通常全部被判 0，导致 **memory_list 为空**（日志里也有 `Filtered to 0 high-information messages`）
  - 随后本项目严格模式在 [reme_unified_memory.py:record_to_memory](file:///d:/SuperAntAgent/memory/reme_unified_memory.py#L370-L489) 强制要求 memory_list 非空，否则抛 `ValueError: memory_list 为空，已拒绝写入。`
- 该异常一路冒泡回 UI；而 UI 的 for 循环没有按角色隔离异常，导致 **后续角色不再处理**：[chroma_admin.py:_init_db_stepwise](file:///d:/SuperAntAgent/ui/chroma_admin.py#L552-L574)

## 具体改动计划
1) 管理端初始化改为调用服务端 `/init`
- 将 [chroma_admin.py:_on_init_db](file:///d:/SuperAntAgent/ui/chroma_admin.py#L514-L536) 的后台线程逻辑，从 `_init_db_stepwise()` 改成调用 `RemoteVectorStoreClient.init_db(preload_system_prompts=preload)`，并把返回的 `created` 列表打印出来。
- 好处：避免错误地把 system prompts 走 personal memory 的 InfoFilter 流。

2) 初始化过程“单角色失败不阻塞后续”
- 修改服务端 [service.py:VectorDbAdmin.init_db](file:///d:/SuperAntAgent/chromaserver/service.py#L342-L406)：
  - 目前 `except: ... raise` 会导致遇到一个角色失败就终止。
  - 改为收集 per-role 的 ok/error，继续处理下一个角色，最终返回 `{ok, created, failed, cost_ms}`。
- 同步让管理端 UI 能展示每个角色失败原因（而不是整个初始化失败）。

3) 角色卡片抽取与写入策略对齐
- 继续复用服务端 `_extract_role_card_blocks()` 的抽取规则（只取 `# 角色定位/# 禁止行为`）：[service.py:_extract_role_card_blocks](file:///d:/SuperAntAgent/chromaserver/service.py#L492-L505)
- 写入 memory_type：保持与该角色 `default_memory_type` 一致（当前服务端已这么做）。

4) 嵌入模型可用性校验（避免后续 role_card upsert 失败）
- 结合日志里 `Unsupported model qwen3-vl-embedding for OpenAI compatibility mode`，检查 `reme_config.yaml` 的 embedding 后端与实际服务端是否匹配；保证 embeddings 接口能对 `qwen3-vl-embedding` 成功返回向量。

## 验证方式
- 运行一次初始化（preload_system_prompts=true），确认：
  - 返回 created 数量等于角色数
  - `_logs/chromaserver_events_*.jsonl` 中 role_card_record 全部为 ok
  - 再次点击初始化应幂等（不重复写入或可接受重复但不报错）
- 通过 `/info` 确认 collections 与 count 发生变化且 workspace_id 正确。
